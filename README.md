### SCORE SUBMISSION PIPELINE

`POST /api/scores/submit` runs these checks in order and short-circuits on the first failure:

```
 1  Method must be POST
 2  Content-Type: application/json required
 3  Body must be a valid JSON object
 4  name and score fields must be present
 5  score must be a finite integer
 6  IP rate limit в”Ђв”Ђв”Ђ 3 requests/minute per IP (in-memory, resets on cold start)
 7  Score bounds в”Ђв”Ђв”Ђв”Ђ 10 в‰¤ score в‰¤ 4500 (game physical max + 20% buffer)
 8  Name sanitization вЂ” uppercase, alphanumeric + space, max 8 chars
 9  Name validity в”Ђв”Ђв”Ђ not all-same-character, not in blocked list
10  Profanity filter в”Ђ checked against built-in word list
11  Session token в”Ђв”Ђв”Ђ signature + expiry + one-time use (skipped if GAME_SECRET not set)
12  Global DB rate в”Ђв”Ђ max 30 inserts/minute across all IPs (circuit breaker)
13  Duplicate guard в”Ђ same name + score in last 30 s в†’ silent 200, no insert
14  Supabase insert в†’ return updated top-10 leaderboard
```

### SESSION TOKEN SYSTEM

Prevents fake scores from being submitted without actually playing the game.

```
GAME STARTS
  в””в”Ђв–¶ POST /api/session/start
        в”њв”Ђв”Ђ Generates UUID session ID
        в”њв”Ђв”Ђ Signs it with HMAC-SHA256 using GAME_SECRET
        в”њв”Ђв”Ђ Persists to game_sessions table (used: false, expires in 30 min)
        в””в”Ђв”Ђ Returns { token: "<base64url>.<hmac-sig>" }

GAME ENDS  (win or game over)
  в””в”Ђв–¶ sessionStorage stores: token + score hash

PLAYER SUBMITS SCORE  (End screen, on Enter)
  в””в”Ђв–¶ POST /api/scores/submit { name, score, sessionToken, scoreHash }
        в”њв”Ђв”Ђ Verify HMAC signature в”Ђв”Ђв”Ђ 401 INVALID_TOKEN if wrong
        в”њв”Ђв”Ђ Check expiry (30 min) в”Ђв”Ђв”Ђ 401 TOKEN_EXPIRED if past
        в”њв”Ђв”Ђ Look up session in DB в”Ђв”Ђв”Ђ 401 SESSION_NOT_FOUND if missing
        в”њв”Ђв”Ђ Check used=false в”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђ 401 SESSION_ALREADY_USED if consumed
        в”њв”Ђв”Ђ Mark session used=true (atomic one-time consumption)
        в””в”Ђв–¶ Insert score
```

| ATTACK | PROTECTED |
|--------|-----------|
| `curl` with arbitrary score | вњ“ вЂ” no valid token |
| Replaying the same token | вњ“ вЂ” marked used after first submit |
| Expired token | вњ“ вЂ” 30-min server-side TTL |
| Forged token signature | вњ“ вЂ” HMAC-SHA256, secret server-only |

> **Dev mode:** if `GAME_SECRET` is not set, token verification is skipped with a console warning. The game works fully without it.

### SCORE HASH CHAIN

During gameplay, `src/utils/scoreHash.ts` builds a hash chain over every score increment:

```
hashв‚Ђ = '0'
hashв‚™ = djb2(hashв‚™в‚‹в‚Ѓ + ":" + points + ":" + source + ":" + total)
```

### CLIENT-SIDE DATA LAYER

`src/services/leaderboard.ts` is the single abstraction over all score I/O:

- `getScores()` вЂ” async, hits `/api/scores`, caches to localStorage on success, falls back to cache on error
- `saveScore(name, score, meta)` вЂ” saves locally first (sync, never lost), then posts to the API with `sessionToken` and `scoreHash` from `meta`
- `getLocalScores()` вЂ” sync read from `invadertony_scores_local` localStorage key
- `LeaderboardError(code)` вЂ” thrown for user-visible API rejections; EndPage catches it and shows the message with a shake animation

### SUPABASE SETUP

Run in the Supabase SQL editor:

```sql
-- Scores table

  id         uuid DEFAULT gen_random_uuid() PRIMARY KEY,
  name       text NOT NULL,
  score      integer NOT NULL,
  created_at timestamptz DEFAULT now()
);
CREATE INDEX scores_score_idx ON scores (score DESC);

  id          uuid DEFAULT gen_random_uuid() PRIMARY KEY,
  session_id  uuid NOT NULL UNIQUE,
  issued_at   timestamptz NOT NULL,
  expires_at  timestamptz NOT NULL,
  used        boolean DEFAULT false NOT NULL,
  created_at  timestamptz DEFAULT now()
);
```

### CLEANUP CRON

`GET /api/cleanup` deletes `game_sessions` rows with `expires_at` older than 2 hours.

- Runs daily at midnight UTC вЂ” configured in `vercel.json в†’ crons`
- Protected by `Authorization: Bearer <CRON_SECRET>` вЂ” Vercel injects this automatically
- Returns `401` for any unauthenticated request

- **Object pooling** вЂ” bullets (30 player + 30 enemy), particles (200), shockwaves (5) are all pre-allocated. Zero `new` calls in the game loop.
- **Pre-allocated vectors** вЂ” `_pA` and `_pB` (`THREE.Vector3`) are module-level singletons reused for every collision check.
- **Shared textures** вЂ” all `basic` enemies share one texture instance; all `elite` enemies share another. Disposed together via `disposeInvaderResources()`.
- **Delta-time capped** вЂ” clamped to 50 ms per frame to prevent the spiral-of-death on tab focus restore.
- **Background renderer pauses** during gameplay вЂ” the game scene has its own embedded starfield, so the background canvas stops its rAF loop to save GPU bandwidth.
- **Asset preloading** вЂ” JS chunks, sprites, and background music are all preloaded at startup behind the loading screen. The game starts with everything in memory.
